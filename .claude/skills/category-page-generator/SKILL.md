# Добавление работы и генерация страниц

## Источник истины
`data/categories.js` и `data/works.js` — единственное место, откуда берётся контент.
`category-{slug}.html`, `work-{slug}.html` и блок DIRECTIONS в `index.html` —
сгенерированный output. НИКОГДА не редактировать их напрямую: правка будет
перезатёрта следующим `node build.js`.

## Алгоритм добавления работы
1. Убедиться, что ассеты (обложка + галерея) лежат в
   `assets/img/{category-slug}/{work-slug}/`. Если файлов нет — не выдумывать
   плейсхолдеры молча, сообщить об отсутствии и остановиться.
2. Добавить один объект в `data/works.js` по схеме:
   `title, slug, category, summary, cover, client, task, role, idea, gallery[], outcome`.
   Категория (`category`) должна быть одним из существующих slug из
   `data/categories.js` (identity, uiux, ads, smm, print, infographics, motion) —
   новые категории не изобретать.
3. Запустить `node build.js`.
4. Показать `git diff --stat`. Ожидаемо затронуты: `category-{category}.html` (в
   которую попала работа), новый `work-{slug}.html`, и, возможно, блок между
   `<!-- DIRECTIONS:START -->` / `<!-- DIRECTIONS:END -->` в `index.html` (если
   изменился count или порядок панелей). Если diff задевает что-то за пределами
   этого списка — остановиться и разобраться, не коммитить.
5. Не коммитить самостоятельно без явной просьбы — только подготовить и показать
   diff.

## Правило сортировки панелей (не пересчитывать вручную — уже в build.js)
По числу работ в категории по убыванию, при равенстве — по алфавиту. Если count
после генерации выглядит не так, как ожидалось — проблема в `build.js`, а не в
том, что нужно поправить `index.html` руками.

## Effort
Задача механическая: заполнить объект по известной схеме и запустить существующий
скрипт — без архитектурных решений. `effort: low` уместен. Если требуется
изменить сам `build.js` или `templates/*.html` (не просто добавить данные) — это
уже архитектурная работа, для неё стоит явно попросить `/effort medium`.
