# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Static, no-build portfolio site for a graphic designer (branding/identity/logos, "CLEVERNOX"). Plain HTML/CSS/JS — no framework, no bundler, no package.json. Content and language is Russian (`lang="ru"`).

`README.md` is the original Russian-language spec/brief for the site (page map, UX principles, visual language, file scheme). Treat it as the design brief this codebase implements — read it for the *why* behind structural decisions (e.g. why work pages are separate files rather than templated).

## Running / previewing

No build step. Open any `.html` file directly, or serve the directory with any static server, e.g.:

```
python3 -m http.server 8000
```

There is no test suite, linter, or CI config in this repo.

## Architecture

**Multi-page static site, one real file per URL** (Approach A from README.md §6 — chosen deliberately over a JS-templated single-page approach for simplicity/SEO/no-JS-required).

- `index.html` — home page: hero, "Мои направления" (category grid), short about, footer.
- `works.html` — flat grid of all case studies.
- `category-{slug}.html` (ads, identity, infographics, motion, print, smm, uiux) — one page per work direction, each either showing a `.work-grid` of case cards or an `.empty-state` block if no work exists yet in that category.
- `work-{slug}.html` — one page per case study (currently `argo`, `fabrika-12`, `kedr`), built from a single repeated template (see below).
- `about.html` — bio page with CV download.
- `css/style.css` — the only stylesheet, plain CSS with `:root` custom properties (no preprocessor).
- `js/main.js` — two independent IIFEs: burger-menu toggle, and scroll-reveal via `IntersectionObserver` (progressive enhancement — `html:not(.js) .reveal` shows content immediately if JS is disabled, and `<html>` only gets the `.js` class via an inline `<script>` in `<head>`).
- `assets/img/` — SVG placeholder art (covers, mockups, sketches) generated by `generate_assets.py`; `assets/cv.pdf` is the downloadable CV.

### The HTML template pattern

Every page repeats the same structural skeleton (copy-paste, not includes — there is no templating engine):

1. `<head>` boilerplate: charset, inline `document.documentElement.classList.add("js")`, viewport, title, description, favicon, Google Fonts preconnect/link (Fraunces + IBM Plex Mono + Inter), `css/style.css`.
2. `<header class="site-header">` with `.nav` — logo link, burger button, `.nav-links` (Работы / Обо мне / Контакты). Nav links to in-page sections (`#directions`, `#contact`) are prefixed with `index.html` when not already on the home page. Current-page link gets `aria-current="page"`.
3. `<main>` with page-specific `<section class="container ...">` blocks. Sections that should animate in on scroll get the `reveal` class.
4. `<footer class="site-footer" id="contact">` — identical on every page: email, Telegram, Behance link, CV download, nav links.
5. `<script src="js/main.js">` at the end of `<body>`.

When adding a new page, copy the closest existing page of that type (e.g. copy `category-ads.html` for a new empty category, or `work-kedr.html` for a new case study) rather than writing from scratch, to keep markup/classes consistent.

### Work/case-study data model

Each case study (`work-{slug}.html`) follows the schema described in README.md §6: `title`, `slug`, `type`, `summary`, `cover`, `client`/`task`/`role` (Контекст), `idea` text with process figures (Идея и подход), `gallery[]` (Результат), `outcome` (Итог), plus prev/next links in `.case-nav` at the bottom for two-way navigation between projects (never a dead-end page — README.md §4).

Adding a new case study means:
- Creating `work-{new-slug}.html` from an existing one, updating all content blocks and image paths.
- Adding a card for it in the relevant `category-{slug}.html` `.work-grid` (or replacing that category's `.empty-state` block if it was previously empty) and in `works.html`.
- Updating `data-count` / `data-empty` and panel ordering on `index.html`'s `.directions-grid` (panels are manually ordered by work count, descending, then alphabetically — see the HTML comment above that grid).
- Fixing up `.case-nav` prev/next links on the new page *and* on its neighbors, since navigation between cases is a manually-maintained chain, not generated.

### Asset generation

`generate_assets.py` (Python, no dependencies beyond stdlib) generates the placeholder SVG artwork under `assets/img/` — covers, mockup variants (business card, sign, packaging, screen), and "before/after" sketch pairs per project slug, plus the about-page avatar placeholder. It writes to a hardcoded `/home/claude/portfolio/assets/img` path (from the environment it was originally authored in) — update `OUT` before rerunning locally. These are intentionally generic geometric placeholders, not real client work; real photography/mockups are expected to replace them (`avatar.png`/`avatar2.jpg` in `assets/img/` are already real photos superseding the generated `avatar.svg`).

### CSS conventions (`css/style.css`)

- Color/typography/spacing tokens are CSS custom properties in `:root` (`--paper`, `--ink`, `--cobalt` accent, `--font-display` (Fraunces) / `--font-body` (Inter) / `--font-mono` (IBM Plex Mono), `--container`, `--radius`).
- Recurring motif: the `.regmark` SVG — a clover of 4 filled hearts, tips pointing to center — used as a decorative icon throughout `.eyebrow` labels and image corners. It's a filled `currentColor` shape, so it inherits blue (`--cobalt`) in `.eyebrow` context; corner variants (`.regmark.tl`/`.regmark.br`) and the work-card thumbnail variant explicitly override to white for contrast over photos.
- Section-level CSS is organized in large comment-delimited blocks per page area (header/nav, hero, directions grid, work grid, case page, about, footer) — when editing styles for one page section, find its block via the banner comments rather than searching by selector alone.
- Mobile breakpoints are per-component (`@media (max-width: 760px)` etc.), not global — check the bottom of each section's CSS block for its own responsive overrides.
