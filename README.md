# Портфолио графического дизайнера — техническое задание

Документ для самостоятельной разработки на HTML / CSS / JS.
Бэкенд не требуется — сайт статический.

---

## 1. Идея и задача

Персональный сайт-портфолио графического дизайнера (брендинг, айдентика, логотипы).
Сайт решает две задачи одновременно:

- **Работодатель** (арт-директор, HR) — оценивает диапазон работ и мышление за ними. Ему важны раскрытые кейсы «задача → идея → результат» и возможность скачать CV.
- **Клиент на фриланс** — оценивает результат и удобство связи. Ему важны сильные работы в контексте и быстрый контакт.

Ядро для обоих одно — сильные кейсы. Различие только в двух точках входа: кнопка «Написать» (клиент) и кнопка «Скачать CV» (работодатель).

Ключевой принцип: **сам сайт — это часть портфолио**. Для графического дизайнера вёрстка, типографика и отступы читаются как проба навыка, поэтому качество исполнения сайта критично.

---

## 2. Архитектура сайта (карта страниц)

Многостраничная структура с настоящими URL (не одностраничник с якорями), чтобы каждый кейс имел свой адрес, который можно дать ссылкой.

```
/                     Главная — витрина
/works.html           Работы — сетка всех проектов
/work-{slug}.html     Страница отдельного проекта (по шаблону, одна на проект)
/about.html           Обо мне (можно оставить блоком на главной)
```

Отдельная страница «Контакты» не нужна — контакты живут в футере на всех страницах.

### Верхнее меню (одинаковое на всех страницах)

Слева — имя/логотип (ссылка на главную). Справа — пункты: **Работы**, **Обо мне**, **Контакты** (якорь на футер).
Меню фиксированное или обычное — на выбор; на мобильном сворачивается в бургер.

---

## 3. Страницы: структура блоков

### 3.1. Главная (`/`)

Короткая витрина, задача — заинтересовать и увести в «Работы».

1. **Первый экран (Hero).** Имя крупно, роль одной строкой («Графический дизайнер: брендинг, айдентика, логотипы»), одна главная кнопка («Смотреть работы»). Для брендинга сильно работает крупная типографика или одна лучшая работа на весь экран вместо абстрактной картинки.
2. **Избранные работы.** 3–4 лучших проекта тизером (превью + название). Кнопка «Все работы» → `/works.html`.
3. **Обо мне кратко.** 2–3 предложения + фото. Ссылка «Подробнее» → `/about.html`.
4. **Футер** (контакты).

### 3.2. Работы (`/works.html`)

1. Заголовок страницы.
2. **Сетка проектов.** 2–3 колонки на десктопе, 1 на мобильном. Карточка = превью-мокап + название + подзаголовок с типом работы (айдентика / логотип / упаковка). Вся карточка кликабельна → страница проекта.
3. (На будущее, при большом числе проектов) фильтры по типу над сеткой: Всё / Айдентика / Логотипы / Упаковка. На старте при 5–7 проектах фильтры не нужны.
4. Футер.

### 3.3. Страница проекта (`/work-{slug}.html`) — ядро сайта

Единый шаблон, наполняемый для каждого проекта. Структура сверху вниз:

1. **Шапка кейса.** Название проекта, одна строка сути (клиент / задача), крупное главное изображение.
2. **Контекст.** Клиент, задача, что требовалось, роль дизайнера. Первое, что ищет работодатель.
3. **Идея и подход.** Текст о логике решения: что пробовала, почему выбрала это направление. Блок, отличающий кейс от галереи. Можно чередовать текст с картинками процесса (эскизы, варианты логотипа).
4. **Результат (галерея карточек).** Сетка изображений финального результата. Для брендинга — мокапы в контексте: логотип на визитке, вывеске, упаковке, в цифре. Крупные изображения, чередование полноширинных и парных.
5. **Итог.** Короткое завершение + результат для клиента, если известен.
6. **Навигация между проектами.** Ссылки «Назад к работам» и «Следующий проект» — чтобы страница не была тупиком.
7. Футер.

### 3.4. Обо мне (`/about.html`)

Текст от первого лица, инструменты (Illustrator, Photoshop, InDesign, Figma), опыт, живое фото. 3–4 предложения без биографии на страницу. Кнопка «Скачать CV» (PDF).

### 3.5. Футер (на всех страницах)

Почта и Telegram крупно и кликабельно, ссылка на Behance, кнопка «Скачать CV», копирайт.

## 4. UX-методы и принципы

- **Правило 3 секунд.** С первого экрана за 3 секунды понятно, кто перед вами и что делает.
- **Двусторонняя навигация.** Со страницы проекта всегда легко вернуться в «Работы» и перейти к следующему кейсу. Никаких тупиковых страниц.
- **Прогрессивное раскрытие.** Главная даёт тизеры → «Работы» дают обзор → страница проекта раскрывает глубину. Пользователь сам выбирает уровень погружения.
- **Одна главная кнопка на экран.** Не заставляйте выбирать между пятью действиями. Один явный призыв, остальное — вторично.
- **Работы в контексте.** Логотип на белом фоне продаёт хуже, чем тот же логотип на мокапе визитки/вывески/упаковки. Всегда показывать применение.
- **Меньше, но сильнее.** 5–7 отличных проектов лучше 15 средних. Слабая работа тянет вниз всё портфолио.
- **Единый шаблон кейса.** Собирается один раз, дальше наполняется. Экономит время и держит сайт целостным.
- **Mobile-first проверка.** Значимая доля просмотров — с телефона. Кривая адаптация у дизайнера читается как непрофессионализм. Проверять каждую страницу на узком экране.
- **Доступность.** Контраст текста к фону не ниже WCAG AA, у всех изображений — атрибут `alt`, навигация с клавиатуры, размер кликабельных зон не меньше 44×44 px.

## 5. UI / визуальный язык

- **Много воздуха.** Крупные отступы, работы «дышат». Пустое пространство — признак вкуса, а не пустота.
- **Ограниченная палитра.** Нейтральная база (чёрный / белый / серый) + один акцентный цвет. Основной цвет дают сами работы.
- **Типографика.** Одна–две гарнитуры с характером. Для брендолога типографика на сайте — заявление о навыке. Чёткая иерархия: крупный заголовок, средний подзаголовок, спокойный основной текст. Ограничить ширину абзаца ~60–75 символов для читаемости.
- **Сетка.** Единая модульная сетка, одинаковые вертикальные ритмы между блоками.
- **Крупные превью работ** и аккуратная сетка карточек.
- **Сдержанные анимации.** Плавное появление блоков при скролле, лёгкий hover на карточках. Без перебора — анимация обслуживает контент, а не наоборот.
- **Единый стиль изображений.** Все мокапы желательно в одной подаче (свет, ракурс, фон), чтобы сетка выглядела цельно.

## 6. Техническая реализация

Сайт статический — бэкенд не нужен, Python не требуется. Хостинг — GitHub Pages / Netlify / Cloudflare Pages (бесплатно).

### Стек

- **HTML** — семантическая разметка (`header`, `nav`, `main`, `section`, `article`, `footer`).
- **CSS** — один файл `style.css`. Раскладка на CSS Grid (сетки карточек) и Flexbox (меню, футер). Переменные `:root` для цветов, шрифтов, отступов. Медиазапросы для мобильной версии.
- **JS** — минимум: бургер-меню, анимации появления при скролле (IntersectionObserver), фильтры работ (если появятся). Без фреймворков на старте.

### Как организовать страницы проектов (два подхода)

**Подход А — отдельные HTML-файлы (проще для старта, надёжнее).**
Каждый проект — свой файл `work-{slug}.html`, сделанный по единому шаблону. Плюс: работает без JS, легко хостить, хорошо для SEO. Минус: при правке шаблона надо обновлять все файлы.

**Подход Б — один шаблон + данные (меньше дублирования).**
Один файл `work.html`, данные всех проектов — в `projects.js` (или `projects.json`). При открытии страницы JS читает `?id={slug}` из URL и подставляет контент. Плюс: одна точка правки. Минус: без JS страница пустая, чуть хуже для SEO.

Рекомендация для «завайбкодить»: **начать с подхода А** — предсказуемее и без подводных камней. Перейти на Б, когда проектов станет много.

### Структура файлов

```
/
├── index.html
├── works.html
├── about.html
├── work-{slug}.html        (по одному на проект)
├── css/
│   └── style.css
├── js/
│   └── main.js
├── assets/
│   ├── img/                (превью, мокапы, фото)
│   └── cv.pdf
└── favicon.ico
```

### Данные проекта (единая схема для каждого кейса)

Поля, которые заполняются для каждого проекта, чтобы шаблон был единообразным:

- `title` — название проекта
- `slug` — часть URL (латиница, через дефис)
- `type` — тип работы (айдентика / логотип / упаковка)
- `summary` — одна строка сути
- `cover` — путь к главному изображению
- `client`, `task`, `role` — блок «Контекст»
- `idea` — текст «Идея и подход»
- `gallery[]` — массив изображений результата (мокапы)
- `outcome` — текст «Итог»

## 7. Порядок работы

1. Собрать 5–7 сильных проектов и подготовить изображения (превью + мокапы в одном стиле).
2. Заполнить схему из раздела 6 для каждого проекта.
3. Свёрстать `index.html` и определить CSS-переменные (цвета, шрифты, отступы) — это фундамент.
4. Свёрстать шаблон страницы проекта на одном кейсе, довести до идеала.
5. Размножить шаблон на остальные проекты.
6. Свёрстать `works.html` (сетку) и `about.html`.
7. Проверить мобильную версию всех страниц.
8. Проверить доступность (контраст, `alt`, навигация с клавиатуры).
9. Опубликовать на статическом хостинге.

---

## 8. Что учесть

- Имя дизайнера должно стоять крупно на первом экране — это часть личного бренда.
- Все мокапы желательно выдержать в едином стиле подачи.
- Каждая страница проекта обязана иметь выход (назад / следующий проект).
- Мобильная версия страницы кейса — вертикальная лента с крупными картинками, комфортная для листания.
